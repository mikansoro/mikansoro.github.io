Working from a plane, without your work computer

2024-01-17
#vpn#work#remote_work#software

Disclaimer

Consider checking with your company's Security team before attempting to work remotely like illustrated in this post. I'm not responsible if you wind up in hot water over anything posted herein.

While this post's example provides and illustrates precautions and protections taken to minimize potential security vectors, the information presented here should be taken as an illustrative thought exercise.

Some employers require you to notify the IT department when traveling with a company-issued device, espeically when traveling overseas. The concept of "work from anywhere" in the post-pandemic world is here to stay; however, employers are (rightfully) weary of machines posessing potentially sensitive data traveling to unsavory places. Embarking on a weekend getaway early on a Friday or getting a high priority page for "something on you can fix" while in the middle of a pre-approved "Avoid Jira for a Few Days" stint shouldn't necessitate lugging another device along. In fact, it doesn't.

This post's focus is on connecting back to a work machine securely over the internet in time of need from another device. Primarily, the post will focus on initiating this connection from inflight wifi, a notoriously high-latency and low-bandwidth environment that makes any form of work difficult—especially remote control of a machine over the internet. However, that should not negate the fact that this configuration will work anywhere provided one can initiate the remote connection. The list of optimizations herein are also non-exhaustive, and are a result of my own trials and troubleshooting.

Background#

I work from a company issued Macbook Pro, that sits connected to a docking station on my desk at all times. The dock provides both power and an ethernet connection to one of my home networks.

Personal VPN#

When roaming away from home, each of my devices are configured with a Wireguard-based VPN, allowing connectivity to several of my home networks. Wireguard offers several benefits over competitors like OpenVPN, offering reduced connecton overhead, latency, and the ability for clients to roam between networks dynamically without dropping connections. The actual setup of this VPN is outside the scope of this post, in this case any VPN that allows connectivity between the device being used and the network of the work device is sufficient.

Important to note that as part of this example, I will not be configuring my personal Wireguard VPN on my work machine. That would be a violation of security policy, and isn't necessary due to the fact that the VPN tunnel has direct access to the network that the work machine is connected to.

Rationale#

While seemingly inconvenient, there's a specific reason I have to connect in the manner I'm describing in this post. Our particular corporate VPN can only be installed on a company-issued device as a matter of security policy. It would be far easier—and less error-prone—to temporarily connect a personal device to the corporate VPN and access internal resources. This introduces all sorts of headaches for myself and my colleagues on the security team:

  • I don't have access to my work credentials on my personal device (and I don't want to)
    • SSH keys, cloud provider authentication, password manager, etc.
  • My personal device has different configurations and/or unsupported toolchains
    • Wrong terraform version, personal vs. work gitconfig, etc.
  • Git repos have to be recloned and stored on a personal device
    • That can be a long process on in-flight wifi, with sufficiently large repositories. And yes, I mean large repositories
    • This necessitates storage of company assets/intellectual property on a personal device, which should be avoided

Constraints, and Working Around Them#

As mentioned above, my work's corporate VPN can only be installed on a company-issued device. In addition, connecting to the VPN requires an OIDC challenge to be completed with our identity provider (IdP). This challenge is browser based, and due to its nature it initiates a request to a localhost callback URL served by the VPN client, passing along the authentication token and any other additional required info. This can't be worked around in a SSH session (that I've found, at least), so VNC is required here to connect to the desktop session for authenticating to the corporate VPN.

Our credentials service also uses a similar paradigm, but due to the need for it to run on headless remote development workstations, it's capable of outputting the challenge URL for the challenge to be completed in a remote browser. This means we're not bound by VNC for issuing temporary credentials, and they can be issued over SSH.

With the above in mind, Before leaving on a trip I enable the following settings on the work machine:

  • ssh server service (under System Preferences > Sharing)
  • vnc service (under System Preferences > Sharing)
  • no sleep mode (via a "caffeinate"-type app, power settings, etc. I normally use KeepingYouAwake)

I'm also a CLI-heavy user, and operate primarily in Emacs with Vim bindings (Evil mode) as my editor of choice. Terminal emacs with vim bindings doesn't translate well over an ssh connection (at least, not with the linux terminal emulator on my personal machine), but I can work entirely through Vim if needed. This allows me to connect from slow, high-latency networks like in-flight wifi and still get things done.

Setup#

Preparation & Optimization#

Ideally before boarding the plane, VNC into the work machine and connect to the corporate VPN. VNC doesn't play nice with most high-latency wifi networks, so it's best to handle this ahead of time. This saves a lot of headache later on. If you have a windows machine and can use Windows Remote Desktop, it's much better at handling transactions over slow networks.

There's another setting we can enable on macos to help stabilize the connection to the remote machine. In order to mac macos respond more reliably in poor connectivity conditions, set the following sysctl option. Full details can be found on this StackExchange answer, but the gist is by default macos will batch sending ACKs until some threshold is met for there being "enough" data to be "worth sending". We want ACKs to be sent immediately to us.

sudo sysctl -w net.inet.tcp.delayed_ack=0

Connection Time#

Now we're ready to actually connect to our work machine and get something done. There are two options here: if your network will sustain it, connect to the desktop via VNC and work normally, or connect over ssh if it doesn't. To utilize our full environment effectively over ssh, we're going to create two different ssh connections.

Those who have used ssh with bad internet know that it can be quite painful, so I actually recommend first installing mosh on both machines and using that to connect instead. This way, any network disruptions won't hang your terminal or force you to start a new session, and your inputs will be buffered to the remote machine when the network is available.

Once connectivity is established via ssh/mosh, setup a separate ssh tunnel and open a chromium/chrome/firefox instance using the tunnel as a socks proxy. With this proxied browser, we can access any internal sites on the corporate network, complete authenticaiton challenges, and more, as all traffic inside the browser act as if its coming from the remote machine.

ssh -ND 1080 -fC user@host
chromium-browser --proxy-server="socks5://localhost:1080"

Conclusion#

There are definitely more optimizations that could be made here, and this just scratches the surface. That being said, I was able to successfully work from a metal tube over the Pacific Ocean via this message in an emergency, in a situation where I didn't have the right computer for the job. If you have more optimizations or suggestions, please reach out. I'd love to hear them.
